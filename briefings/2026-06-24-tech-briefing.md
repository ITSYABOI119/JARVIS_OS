# JARVIS Daily Tech Briefing — Wednesday, June 24, 2026

**Lead — Linux's 8250/16550 UART driver is getting a modernization pass in 7.2 (TTY/serial pull merged ~Jun 22–23).** Greg KH's serial pull for the open 7.2 merge window reworks the legacy 8250 driver — which also drives 16450/16550 ports — adding console flow control, fixing a possible soft-lockup, plus cleanups to make it "more modern." *Why it matters:* JARVIS hand-rolls its own 16550A driver (`uart_16550.c`) and routes every on-screen line through `putc_serial` → the NVMe log mirror, so the serial path is load-bearing for your whole "verify-UI-from-the-log" workflow. Mainline's reference 16550 is the canonical impl to diff against — the soft-lockup fix and flow-control handling are exactly the corner cases a from-scratch driver tends to get wrong. ## Systems / OS

- **A tiny move that reclaims memory-write bandwidth: iomap drops a redundant memset, +~5% IOPS (7.2).** Fengnan Chang's change skips the `memset` of the iomap struct in `iomap_iter()` once iteration is done; in 4K-randread NVMe-polling-via-io_uring that memset was burning memory write bandwidth for nothing, and removing it lifts ext4/XFS IOPS ~5%. *Why it matters:* your Phase-4 goal-#1 conclusion is that inference is memory-bandwidth-bound (~2.9 GB/token), and this is a clean reminder that the same discipline — kill redundant writes in the hot path — is where the wins are; doubly relevant since your NVMe model-loader is polled. [Phoronix](https://www.phoronix.com/news/Linux-7.2-IOmap-EXT4-XFS) · [xda](https://www.xda-developers.com/someone-moved-three-lines-of-code-in-linux-72-and-got-a-5-storage-speed-boost/)
