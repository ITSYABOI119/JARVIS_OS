# JARVIS Daily Tech Briefing — Friday, June 26, 2026

**Lead — Update: Cache-Aware Scheduling gets a follow-on patch series that goes *multi-LLC* (Jun 25).** A week after Cache-Aware Scheduling merged for Linux 7.2 (covered Jun 18), Hygon posted patches extending it from a single-LLC task-aggregation model to **hierarchical, topology-aware aggregation** that dynamically expands/contracts across scheduling domains — fixing the case where a workload spills past one LLC and locality is lost. Benchmarks: up to **360% in MySQL**, 49% Hackbench, 20% Schbench. *Why it matters:* JARVIS's box is a **dual-CCX 2700X** — two 4-core CCXs, each with its own L3 — so `NUM_NODES=6` straddles two LLC domains, exactly the cross-domain case this series targets. You hand-pin PA/PB and the M3 worker pool with `seL4_TCB_SetAffinity` because seL4 won't auto-distribute, so mainline's reasoning about multi-LLC locality is a direct reference for *which* cores the threadpool should land on (keep a worker set within one CCX's L3 vs. spanning both). Caveat: your loop is memory-bandwidth-bound, so this is a placement lever, not a tok/s lever. [Phoronix](https://www.phoronix.com/news/Extended-Cache-Aware-Sched)

## Systems / OS

- **Linux Foundation launches "Akrites" to coordinate fixes for AI-discovered OSS vulns (Jun 25).** A $12.5M, confidentiality-first effort (AWS, Anthropic, Google, Microsoft/GitHub, NVIDIA, Red Hat, Rust Foundation, et al.) with a shared SIRT + standardized CVD process and a "maintainer of last resort" for unmaintained critical packages — aimed at patching AI/LLM-found bugs before they're exploited. *Why it matters:* the same AI-discovered-vuln wave behind your Jun 19 (ARM64 patch-drag) and Jun 24 (AI-assisted `perf.data` hardening) items now has an industry response; JARVIS sits on OSS (seL4, ggml-lineage inference code, llama.cpp as your reference), so upstream coordination touches your supply chain. [Phoronix](https://www.phoronix.com/news/Akrites) · [PRNewswire](https://www.prnewswire.com/news-releases/linux-foundation-and-industry-leaders-launch-akrites-to-defend-critical-open-source-software-against-ai-enabled-cyber-threats-302811165.html)
- **Linux 7.2 MM: batch-free order-0 pages + tighten mmap readahead (Jun 24).** The page allocator now frees a contiguous run of order-0 pages as a *batch* (undoing a 2025 vmalloc perf regression; benefits `vfree()` and frozen-page release), and `mmap_miss` accounting is tightened to cut redundant synchronous readahead I/O on sparse/strided access. *Why it matters:* same "batch the page ops, kill redundant hot-path I/O" discipline that pays off on memory-bandwidth-bound paths — your inference inner loop and the polled NVMe model loader. [Phoronix](https://www.phoronix.com/news/Linux-7.2-MM)
- **EROFS adds sparse-pcluster support for large AI datasets + faster chunk mapping (Jun 23).** Alibaba's Gao Xiang landed code to mark whole pclusters as holes (preserving sparsity so copy-up/`SEEK_HOLE` behave) plus optimized request mapping for chunk-based inodes. *Why it matters:* JARVIS streams a ~2.9 GB GGUF **read-only** off NVMe through a hand-rolled FAT32 path; EROFS is the canonical read-only-FS-for-AI-blobs design (chunk mapping, hole handling) — a reference if you ever outgrow FAT32 for model storage. [Phoronix](https://www.phoronix.com/news/EROFS-Sparse-AI-Datasets)
